tag Determines what version of Plex to install. yes public

version tag (e.g. "1.12.3.4973-215c28d86")		 public For list of version tags, see here.

The beta tag (formerly plexpass) requires an active Plex Pass account.

Hardware Transcoding requires an active Plex Pass account.

If you decide to change the tags later, you will need to update Plex by running the Cloudbox install command with the "plex" tag (i.e. sudo ansible-playbook cloudbox.yml --tags plex).
transcodes Path of temporary transcoding files. yes /home/{{user}}/transcodes Note 1: It is recommended to not use /tmp or /dev/shm as a transcode location because the paths are cleared on reboots, causing Docker to create the folder as root and Plex transcoder to crash. Another reason why not to: https://forums.plex.tv/discussion/comment/1502936/#Comment_1502936.

Note 2: Having {{user}} in the path tells Ansible to fill in the username automatically. You do not need to fill in your actual username.

version Rclone version that is installed by Cloudbox. yes latest
version number (e.g. 1.42)		 latest
backup
tar_dest Path for local backups (.tar). no /home/{{user}}/Backups Only the the most recent backup is kept.

Note 1: Ensure the path does NOT have a trailing slash (/) or else backup will fail (i.e. /sample/path, but not /sample/path/).

Note 2: Having {{user}} in the path tells Ansible to fill in the username automatically. You do not need to fill in your actual username.
rsync_dest Path for rsync backups (.tar). no Only the the most recent backup is kept.

Note: Ensure the path does NOT have a trailing slash (/) or else backup will fail (i.e. /sample/path, but not /sample/path/).
rclone_dest Path for cloud backups (e.g. Google Drive). no google:/Backups Older backups are stored in the archived folder.
use_rsync Option to enable/disable rsync backup. yes true
false		 false If set to true, you must specify rsync_dest.
use_rclone Option to enable/disable cloud backups (e.g. Google Drive). yes true
false		 false If set to true, you must specify rclone_dest.
cron_time How often to backup. no reboot
yearly
annually
weekly
daily
hourly		 weekly Only needed when cron_state is set to present

Note 1: It is not recommended to schedule backups hourly as backing up may take a long time and cause future backup attempts to fail (the backup will not occur while another one is in progress, thanks to backup.lock file being created/removed during this process).

Note 2: This option just allows the script to schedule the backup for you. You can manually schedule cron to run backups with ansible-playbook cloudbox.yml --tags backup called as root.
cron_state Option to enable/disable automatic backups. yes absent
present		 absent If set to present, you must specify cron_time.

absent will remove any existing backup schedule.

present will ensure it is always scheduled.

Note: When this option changed (e.g. absent to present; present to absent), a manual backup must be run once in order to set the backup schedule (i.e. sudo ansible-playbook cloudbox.yml --tags backup).

This option just allows Cloudbox to schedule the backup for you. You can also setup scheduled backups by creating root cron tasks.
pushover_app_token Pushover App Token. no Enables notifications to be sent when a backup task starts and finishes (requires both the pushover_app_token and the pushover_user_key).
pushover_user_key Pushover User Key. no Enables notifications to be sent when a backup task starts and finishes (requires both the pushover_app_token and the pushover_user_key).
